The ATM Playground works offline after the first visit. This is implemented as a Progressive Web App (PWA) using [Workbox](https://developer.chrome.com/docs/workbox/) — a set of libraries that simplify service worker authoring — integrated into Create React App's build pipeline.

---

## Why offline support?

The primary motivation is performance, not connectivity. The `hand_landmarker.task` model file is several megabytes. Without caching, every page load re-downloads it. With the service worker in place:

- **First visit:** Assets are fetched from the network and stored in the browser cache.
- **Subsequent visits:** Assets are served from cache instantly; the network is checked in the background and the cache is refreshed if a newer version is available (`stale-while-revalidate`).
- **Offline:** The app and model file are available from cache even without a network connection.

---

## What gets cached

Workbox precaches everything in the production build manifest (`self.__WB_MANIFEST`), which includes:

- `index.html` (the app shell)
- All compiled JavaScript and CSS bundles
- Static assets in `public/` (PNG, GIF, MP3 audio files)
- **`hand_landmarker.task`** — the MediaPipe model file

A second runtime cache rule covers same-origin files with these extensions at request time:

| Extension | Examples |
|-----------|----------|
| `.task` | `hand_landmarker.task` |
| `.png` | UI images |
| `.jpg` | Photos |
| `.gif` | Animated demos |
| `.mp3` | Sound effects |

## service-worker.js

The full service worker source (`src/service-worker.js`):

```javascript
import { clientsClaim } from 'workbox-core';
import { ExpirationPlugin } from 'workbox-expiration';
import { precacheAndRoute, createHandlerBoundToURL } from 'workbox-precaching';
import { registerRoute } from 'workbox-routing';
import { StaleWhileRevalidate } from 'workbox-strategies';

clientsClaim();

// Precache all assets generated by the build process.
// URLs are injected into __WB_MANIFEST by the Workbox build step.
precacheAndRoute(self.__WB_MANIFEST);

// App Shell routing: all navigation requests return index.html
const fileExtensionRegexp = new RegExp('/[^/?]+\\.[^/]+$');
registerRoute(
  ({ request, url }) => {
    if (request.mode !== 'navigate') return false;
    if (url.pathname.startsWith('/_')) return false;
    if (url.pathname.match(fileExtensionRegexp)) return false;
    return true;
  },
  createHandlerBoundToURL(process.env.PUBLIC_URL + '/index.html')
);

// Runtime caching for same-origin static assets (including .task model file)
registerRoute(
  ({ url }) =>
    url.origin === self.location.origin &&
    (
      url.pathname.endsWith('.png') ||
      url.pathname.endsWith('.mp3') ||
      url.pathname.endsWith('.jpg') ||
      url.pathname.endsWith('.task') ||
      url.pathname.endsWith('.gif')
    ),
  new StaleWhileRevalidate({
    cacheName: 'assets',
    plugins: [
      new ExpirationPlugin({
        maxEntries: 50,
        maxAgeSeconds: 30 * 24 * 60 * 60, // 30 days
      }),
    ],
  })
);

// Allow the web app to trigger skipWaiting
self.addEventListener('message', (event) => {
  if (event.data && event.data.type === 'SKIP_WAITING') {
    self.skipWaiting();
  }
});
```

### Caching strategy explained

**`stale-while-revalidate`** serves the cached version immediately while fetching a fresh copy from the network in the background:

1. Request arrives → cache hit found → respond immediately with cached asset.
2. Simultaneously fetch the asset from the network.
3. If the network response differs, update the cache for the next request.

This means the user never waits for the model file to download on repeat visits, but the cache is still kept fresh.

## serviceWorkerRegistration.js

The registration module (`src/serviceWorkerRegistration.js`) is called from `index.jsx`. It only registers the service worker in **production** builds:

```javascript
export function register(config) {
  if (process.env.NODE_ENV === 'production' && 'serviceWorker' in navigator) {
    const publicUrl = new URL(process.env.PUBLIC_URL, window.location.href);
    if (publicUrl.origin !== window.location.origin) {
      return; // Service worker won't work with a CDN origin mismatch
    }

    window.addEventListener('load', () => {
      const swUrl = `${process.env.PUBLIC_URL}/service-worker.js`;

      if (isLocalhost) {
        checkValidServiceWorker(swUrl, config);
      } else {
        registerValidSW(swUrl, config);
      }
    });
  }
}
```

Key behaviors:

- **Development (`npm start`):** The service worker is **not** registered. You always get fresh assets from the dev server.
- **Production build:** The service worker is registered on `load`. On subsequent visits, the precache kicks in.
- **Localhost:** Extra validation runs to confirm a valid service worker file exists before registering.

## Unregistering the service worker

The module also exports an `unregister` function for troubleshooting:

```javascript
export function unregister() {
  if ('serviceWorker' in navigator) {
    navigator.serviceWorker.ready
      .then((registration) => {
        registration.unregister();
      })
      .catch((error) => {
        console.error(error.message);
      });
  }
}
```

<Note>
  Call `unregister()` instead of `register()` in `index.jsx` to disable the service worker entirely. This is useful when debugging caching issues or when you want to ensure users always receive the latest build. After unregistering, close all tabs and reopen the app to clear the old service worker.
</Note>

## Workbox packages used

All packages are at version `^7.0.0`:

| Package | Purpose |
|---------|----------|
| `workbox-core` | Foundation: `clientsClaim`, `skipWaiting` |
| `workbox-precaching` | `precacheAndRoute`, build-time manifest injection |
| `workbox-routing` | `registerRoute` for runtime route matching |
| `workbox-strategies` | `StaleWhileRevalidate` and other caching strategies |
| `workbox-expiration` | `ExpirationPlugin` for cache size / age limits |
| `workbox-cacheable-response` | Filter responses to cache by status code |
| `workbox-broadcast-update` | Notify clients when a cached response is updated |
| `workbox-background-sync` | Retry failed requests when connectivity is restored |
| `workbox-navigation-preload` | Speed up navigation requests with preload hints |
| `workbox-range-requests` | Support byte-range requests in cached responses |
| `workbox-streams` | Assemble streaming responses from cache + network |

Full Workbox documentation: [developer.chrome.com/docs/workbox](https://developer.chrome.com/docs/workbox/)
